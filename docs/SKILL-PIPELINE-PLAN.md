# 技能自动化流水线方案 v2

> 用户价值优先 — 不是"收集技能"，是"解决用户问题"
> 
> 日期：2026-03-14
> 状态：待旭确认

---

## 核心原则

用户来网站不是找技能列表的——他们是带着问题来的："我想自动发公众号""我想监控竞品""我想让AI帮我管飞书"。

所以流水线的终点不是"上架了X个技能"，而是"又解决了X个真实场景"。

## 一、用户画像（决定抓什么）

| 用户类型 | 典型需求 | 对应场景 |
|---------|---------|---------|
| 自媒体人 | 自动写稿/发布/涨粉 | 公众号、小红书、Twitter运营 |
| 创业者/小老板 | 省人力、自动化重复工作 | 客服、报表、审批、监控 |
| 开发者 | 效率工具、代码review | GitHub、CI/CD、MCP |
| 职场打工人 | 办公提效 | 飞书/钉钉/Notion/邮件 |
| AI爱好者 | 折腾新玩法 | 多Agent、语音、智能家居 |

**抓取策略围绕这5类人的需求，不是见技能就收。**

### 过滤规则（不收的）

- 纯demo/toy项目（README只有3行、无SKILL.md）
- 6个月没更新的死仓库
- 功能跟已有技能高度重复的（除非明显更好）
- 非中文场景且无通用价值的（比如只支持某个小众外国平台）

## 三、安全审查（三道门）

这是差异化——别的技能站没做安全标注。

### 第1道：静态扫描（脚本自动跑）

检查项：
- ❌ 是否读取 `~/.openclaw/` 敏感配置
- ❌ 是否有 `curl/wget` 往外发数据（排除已知安全API）
- ❌ 是否读取 ENV 变量（API key窃取风险）
- ❌ 是否有 base64 编码的可疑内容
- ❌ 是否有 `eval/exec` 动态执行
- ❌ 是否引用已知恶意域名/IP
- ❌ npm/pip 依赖是否在恶意包名单上

**命中任何一项 = 自动拒绝**

### 第2道：权限分级（自动）

| 等级 | 含义 | 示例 | 处理 |
|------|------|------|------|
| 🟢 安全 | 只读本地文件、只输出文本 | weather、video-frames | 自动上架 |
| 🟡 注意 | 读写文件、调用已知API | feishu-im、github | 自动上架+标注 |
| 🔴 高危 | 执行shell命令、访问任意URL、安装依赖 | browserwing、coding-agent | 进人工审核队列 |

### 第3道：信誉评分（自动）

**加分项：**
- ClawHub官方认证 (+50)
- GitHub star>100 (+30)
- 作者有其他正常项目 (+20)
- 被awesome列表收录 (+20)
- 有完整README + SKILL.md (+10)

**减分项：**
- 新建<30天 (-20)
- 匿名作者无profile (-30)
- 无任何star/fork (-20)

**判定：**
- 总分≥30 → 自动通过
- 总分<30 → 进人工审核队列

### 审查结果展示（用户可见）

- ✅ **已验证** — 通过三道门
- ⚠️ **社区推荐** — 来源可信但未深度审查
- 🔒 **权限等级** — 低/中/高

## 四、分类体系

每个技能只归一个主分类 + 可选标签：

| 分类 | 覆盖场景 |
|------|---------|
| 社交与内容 | 公众号/小红书/Twitter/YouTube/内容创作 |
| 办公效率 | 飞书/Notion/日历/邮件/文档 |
| 生活助手 | 天气/提醒/智能家居/记账 |
| 开发工具 | GitHub/代码/部署/MCP/调试 |
| 数据与监控 | 爬虫/竞品/价格/舆情/报表 |

**自动分类方法：** 读SKILL.md的description + tags，用关键词匹配归类。匹配不上的标记"待分类"进人工队列。

## 六、场景套装（核心差异化）

不是"技能A+技能B"的简单拼凑，是完整的解决方案。

### 每个套装包含

1. **场景名** — 用户能秒懂的一句话
2. **解决什么问题** — 用户痛点描述
3. **包含技能** — 3-5个技能，说清每个干什么
4. **安装命令** — 一键复制
5. **配置步骤** — 从零到跑通要几步
6. **真实效果** — 实际使用效果描述
7. **注意事项** — 踩坑点

### 初始10个套装

| 套装 | 面向用户 | 核心技能 |
|------|---------|---------|
| 公众号全自动运营 | 自媒体人 | wechat-publisher + content-research + image-gen |
| 小红书矩阵运营 | 自媒体人 | xiaohongshu-mcp + baoyu-xhs-images |
| Twitter/X出海获客 | 创业者 | baoyu-post-to-x + agent-reach |
| 飞书办公全家桶 | 职场人 | feishu-im + bitable + doc + calendar + task |
| GitHub开发提效 | 开发者 | github + coding-agent + gh-issues |
| 竞品监控预警 | 创业者 | browserwing + agent-reach + cron |
| AI日报自动生成 | 所有人 | web-search + cron + feishu-im |
| 客户跟进自动化 | 小老板 | feishu-bitable + feishu-im + cron |
| 多Agent团队协作 | AI爱好者 | coding-agent + multi-agent |
| 新手入门三件套 | 新手 | weather + github + tmux |

**套装自动更新：** 新技能入库时，003检查它能否替换/补充现有套装。

## 七、完整流水线（分工明确）

```
每天 5:00 Cron触发 → 003开始执行
  │
  ├─ ① 003：抓取（30分钟内完成）
  │   ├─ 抓 ClawHub 最新+热门
  │   ├─ 抓 GitHub trending/awesome
  │   ├─ 抓 Reddit 热帖提到的技能
  │   ├─ 对比 skills-registry.json（现有库）
  │   ├─ 只保留：新增的 + 有更新的
  │   └─ 输出 → pipeline/01-raw.json
  │
  ├─ ② 003：安全审查 + 格式化（20分钟内完成）
  │   ├─ 对每个新技能跑安全扫描脚本
  │   ├─ 权限分级 + 信誉评分
  │   ├─ 自动分类（关键词匹配）
  │   ├─ 生成标准JSON数据
  │   ├─ 检查是否匹配现有套装
  │   ├─ ✅通过的 → pipeline/02-approved.json
  │   ├─ ⚠️待审的 → pipeline/02-review.json
  │   └─ ❌拒绝的 → pipeline/02-rejected.json（记录原因）
  │
  ├─ ③ 003：通知旭（审查完立刻）
  │   ├─ 发Telegram汇总：
  │   │   "今日扫描X个技能，通过Y个，待审Z个，拒绝W个"
  │   └─ 待审技能列表（名+原因+链接，旭回复Y/N）
  │
  ├─ ④ 004：上架部署（003完成后自动触发）
  │   ├─ 读 02-approved.json
  │   ├─ 更新网站数据文件（src/data/skills.ts）
  │   ├─ 构建 + 部署
  │   └─ 通知旭："已上架X个新技能，网站已更新"
  │
  └─ ⑤ 001：监工（心跳时检查）
      ├─ 003有没有在5:30前完成抓取？没完成→推
      ├─ 004有没有在6:00前完成部署？没完成→推
      └─ pipeline目录有没有异常（空文件/格式错误）→报警
```

## 十、风险和兜底

| 风险 | 应对 |
|------|------|
| 003抓取超时/失败 | 001心跳监控，失败自动重试一次 |
| 安全扫描漏判 | 高危技能永远进人工队列，不自动上架 |
| 技能质量参差不齐 | 信誉评分+过滤规则，宁缺毋滥 |
| 分类不准确 | 匹配不上的进人工队列，不硬归类 |
| 004部署失败 | 回滚到上一版本，通知旭 |

---

> **一句话总结：** 003每天抓+审+格式化，004自动上架，001监工，旭只看审核队列和最终效果。
