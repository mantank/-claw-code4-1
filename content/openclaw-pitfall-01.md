# 折腾OpenClaw一整天，踩了哪些真实的坑

深夜12点，我盯着屏幕，刚跑通了最后一个测试。

从下午开始到现在，我一直在折腾OpenClaw——一个可以在本地跑AI助手的开源框架。目标很简单：让它帮我自动写公众号、配图、发小红书，最后一键发布。

听起来不难，实际上一路踩坑。

今天把真实踩过的坑写出来，不是教程，是实战记录。

---

**01 / 小龙虾有两只爪子，我只给了它一只**

OpenClaw支持同时跑多个助手实例。我设了一主一副，主助手（001）负责统筹，副助手（002）负责执行——写稿、配图、发布。

问题来了：002没有消息推送能力。

任务跑完了，002闷头干完活，没法主动告诉我。我得自己去问它有没有完成。

原因很简单：002是独立的OpenClaw实例，需要单独配置Telegram Bot通道，我只给001配了，忘了002。

加上Bot Token，两只爪子都能动了。

---

**02 / 定时任务发消息，发到哪儿去了？**

设置了一个定时任务，每天定时推送X平台上几个大佬的最新动态。

Cron跑了，日志显示成功，消息没来。

查了半天，发现问题在一个参数：`channel: last`。

这个配置的意思是"发到上次用过的频道"。在正常对话里没问题，但Cron是独立跑的，没有"上次"的记录，它找不到该发给谁，就静默失败了。

改成 `channel: telegram`，写死，再也不丢了。

---

**03 / 另一个Cron，API Key没了**

修完上面那个，另一个Cron还是不能用，报错：No API Key。

OpenClaw有多个运行Profile，有点像浏览器的多用户模式。这个Cron指定了`dev` Profile来跑，但dev Profile是个空壳，Key没有复制进去。

把配置文件从主Profile复制一份过去，问题解决。

教训：建了新Profile，记得检查Key有没有带进去。

---

**04 / 小红书发帖，卡死了**

用OpenClaw的小红书MCP服务发帖，命令发出去，进程挂着，没有任何输出。等了三分钟，啥也没发生。

杀进程，重试，一样。

最后发现：发帖必须带图片。

只传了标题和正文，MCP服务在等图片，图片没来，它就一直等。

加上本地图片路径，命令直接跑通。之后写进了铁律：**没有图片，不调发布接口。**

顺便说一句：测试阶段别往真实账号发，踩了这个我不想说细节。

---

**05 / 飞书文档，怎么都读不出来**

有人分享了一个飞书文档，想让小龙虾读取里面的内容。

用BrowserWing（浏览器自动化工具）打开，页面空白。换成直接抓URL，返回一堆JS代码。

飞书文档是动态渲染的，普通抓取拿到的只是HTML壳，内容还没加载。再加上很多文档需要登录才能看。

折腾了一圈，最后发现飞书有开放API，可以直接读文档原始内容，不需要浏览器，不需要对方授权（只要文档是公开的）：

```
GET /open-apis/docx/v1/documents/{doc_token}/raw_content
```

doc_token就是飞书文档URL最后那段字符串。

现在读飞书文档是稳定能力，这个方法写进了工具文档备用。

---

**06 / 图片生成悄悄在烧钱**

之前配置的图片生成用的是某家大模型的图片API。

今天去控制台看了一眼，账户已经扣了一笔钱。每张图将近5毛钱，我根本没意识到一直在花。

换成了国内的另一个图片模型，每张图0.04元，效果差不多，中文文字渲染还更清晰。

同样的图，便宜了接近10倍。

**平台尺寸速查（已经踩过"尺寸不对图片被裁掉"的坑）：**

- 小红书封面（3:4竖版）：1104×1472
- 公众号封面（16:9横版）：1664×928
- PPT配图（16:9）：1664×936
- 正方形配图：1328×1328

**07 / 改别人的工具，一行没改干净**

找到一个开源的PPT自动生成工具，核心逻辑是：给一篇文章，AI分析结构，每页生成一张图，最后打包成可播放的PPT。

把里面的图片模型换成我自己的，运行时报错：

```
NameError: name 'load_dotenv' is not defined
```

我注释掉了`import`，但函数体里还有调用，Python跑到那里直接崩了。

把那个函数整个替换掉，改成空函数，搞定。

教训：改别人代码，全文搜索所有相关引用，不能只改一处。

改完测试了两页PPT，渐变毛玻璃风格，16:9，出图正常。

---

**08 / 小龙虾把自己改挂了**

这是今天最戏剧性的一个坑。

在给002配置消息通道的时候，我让001（主助手）帮忙修改配置文件。它改了，但改错位置了——把自己的配置文件动了。

改完之后，001直接没响应了。

发消息，没回。重新问，还是没回。

001把自己改挂了。

这时候只能从外部介入。登录服务器，用Claude Code（另一个独立的编程工具，不依赖OpenClaw）直接打开配置文件，把被改乱的内容恢复回去，重启服务，001才重新活过来。

教训很明确：**让AI助手修改自己的运行配置，是高风险操作。** 配置文件改之前先备份，或者直接在服务器上手动改，不要让它自己动自己。

以后凡是涉及OpenClaw自身配置文件的修改，我来动，不让它动。
